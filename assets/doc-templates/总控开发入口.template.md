- 项目名称：{项目名称}
- 项目根目录：{项目根目录}
- 开发文档目录：{开发文档目录}

## 启动方式
用户应先复制 `00_新窗口启动提示词.md` 到新的 Codex 窗口。该提示词用于明确当前会话进入主控开发执行阶段，而不是继续生成开发文档。

## 主控身份
读取本文档的当前 vibe coding 智能体必须确认自己是 A 主控智能体，且本项目同时只能有一个 A 主控。

## 执行阶段授权口径
- 文档生成阶段只生成或维护开发控制文档，不修改产品代码。
- 主控开发执行阶段可以在完成 `00_启动前环境核验.md` 后推进产品代码开发。
- A 主控负责协调、进度、日志、Review 编排和最终汇报。
- 实现型 workstream 可作为 worker 子智能体修改权限矩阵中授权的模块，并提交自测证据。
- 契约、Review、审计角色通常只读；除非本文档集明确授权，不直接修改产品代码。
- 未进入权限矩阵的角色不得修改产品代码。

## 执行规则
- A 先完成启动前环境核验，再分派可写开发任务。
- A 维护任务进度。
- A 按 `多智能体开发协作规范.md` 维护项目画像、权限矩阵、子智能体注册表、生命周期和复用策略。
- 真实 Codex 子智能体只有在用户明确要求多智能体、子智能体、委派或并行开发时创建。
- 创建或派发真实 Codex 子智能体时，A 必须优先读取权限矩阵中的 `Recommended reasoning effort`，并在任务说明或调用参数中显式传递或要求 `reasoning_effort`。
- 若目标 workstream 未配置推荐思考程度，才继承父会话默认 `reasoning_effort`，并在任务进度中记录该 fallback。
- 因成本、速度或工具限制需要调整推荐思考程度时，必须先记录原因并获得 A 主控确认。
- 实现型任务优先派给 worker 子智能体；探索型任务优先派给 explorer 子智能体。
- 实现产物必须经过测试 Review。
- 最终交付前必须经过进度一致性审计。
- 子智能体不直接修改最终任务状态。

## 子智能体复用入口
分派任何子任务前，先阅读 `多智能体开发协作规范.md` 的“项目画像”“权限矩阵”“子智能体注册表”“复用策略表”和“创建前决策门”。

## 最终收口
```text
环境核验完成 -> 实现全部提交 -> 最终 Review 通过 -> 最终审计通过 -> A 汇报用户
```
